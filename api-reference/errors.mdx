# Error Handling

The TalkOS API uses conventional HTTP response codes to indicate the success or failure of an API request. In general:

- `2xx` - Success
- `4xx` - Client errors (invalid request, authentication issues)
- `5xx` - Server errors (something went wrong on our end)

## Error Response Structure

All error responses follow a consistent format:

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable error description",
    "details": {
      // Additional context about the error
    }
  },
  "meta": {
    "requestId": "req_abc123def456",
    "timestamp": "2026-02-18T10:30:00Z"
  }
}
```

<Tip>
  Always include the `requestId` when contacting support - it helps us quickly locate and debug issues.
</Tip>

## HTTP Status Codes

### 400 Bad Request

The request was malformed or contained invalid parameters.

<CodeGroup>
```json Validation Error
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": {
      "errors": [
        {
          "field": "phoneNumber",
          "message": "Phone number must be in E.164 format",
          "received": "9876543210"
        },
        {
          "field": "agentId",
          "message": "agentId is required",
          "received": null
        }
      ]
    }
  },
  "meta": {
    "requestId": "req_abc123",
    "timestamp": "2026-02-18T10:30:00Z"
  }
}
```

```json Invalid JSON
{
  "success": false,
  "error": {
    "code": "INVALID_JSON",
    "message": "Request body contains invalid JSON",
    "details": {
      "parseError": "Unexpected token } at position 45"
    }
  },
  "meta": {
    "requestId": "req_abc123",
    "timestamp": "2026-02-18T10:30:00Z"
  }
}
```

```json Missing Required Field
{
  "success": false,
  "error": {
    "code": "MISSING_REQUIRED_FIELD",
    "message": "Required field 'name' is missing",
    "details": {
      "field": "name",
      "type": "string"
    }
  },
  "meta": {
    "requestId": "req_abc123",
    "timestamp": "2026-02-18T10:30:00Z"
  }
}
```
</CodeGroup>

**Common Causes:**
- Missing required fields in the request body
- Invalid field types (string instead of number)
- Invalid phone number format
- Invalid enum values

```json Missing API Key
  },
  "meta": {
```

```json Resource Access Denied
{
  "success": false,
  "error": {
    "code": "RESOURCE_ACCESS_DENIED",
    "message": "You do not have access to this resource",
```

- API key has restricted permissions
- Trying to access another tenant's resources

### 429 Too Many Requests

You've exceeded the rate limit.

```json
{
  "success": false,
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many requests. Please slow down.",
    "details": {
      "limit": 100,
      "window": "60s",
      "retryAfter": 32
    }
  },
  "meta": {
    "requestId": "req_abc123",
    "timestamp": "2026-02-18T10:30:00Z"
  }
}
```

**Response Headers:**

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Maximum requests per window |
| `X-RateLimit-Remaining` | Requests remaining in current window |
| `X-RateLimit-Reset` | Unix timestamp when the limit resets |
| `Retry-After` | Seconds to wait before retrying |

**Solutions:**
- Implement exponential backoff
- Cache responses where possible
- Upgrade to a higher plan for increased limits
- Use bulk endpoints for batch operations

### 500 Internal Server Error

Something went wrong on our servers.

```json
{
  "success": false,
  "error": {
    "code": "INTERNAL_ERROR",
    "message": "An unexpected error occurred. Our team has been notified.",
    "details": {}
  },
  "meta": {
    "requestId": "req_abc123",
    "timestamp": "2026-02-18T10:30:00Z"
  }
}
```

**What to do:**
1. Retry the request with exponential backoff
2. If the error persists, contact support with the `requestId`
3. Check [status.talkos.io](https://status.talkos.io) for any ongoing incidents


## Error Code Reference

| Code | HTTP Status | Description |
|------|-------------|-------------|
| `VALIDATION_ERROR` | 400 | Request validation failed |
| `INVALID_JSON` | 400 | Malformed JSON in request body |
| `MISSING_REQUIRED_FIELD` | 400 | Required field not provided |
| `INVALID_FIELD_TYPE` | 400 | Field has wrong type |
| `INVALID_PHONE_NUMBER` | 400 | Phone number format invalid |
| `MISSING_API_KEY` | 401 | x-api-key header missing |
| `INVALID_API_KEY` | 401 | API key invalid or revoked |
| `MISSING_TENANT_ID` | 401 | x-tenant-id header missing |
| `INVALID_TENANT_ID` | 401 | Tenant ID not found |
| `INSUFFICIENT_PERMISSIONS` | 403 | API key lacks required permissions |
| `PLAN_LIMIT_EXCEEDED` | 403 | Feature not in current plan |
| `RESOURCE_ACCESS_DENIED` | 403 | Cannot access this resource |
| `ACCOUNT_SUSPENDED` | 403 | Account has been suspended |
| `RESOURCE_NOT_FOUND` | 404 | Requested resource not found |
| `ENDPOINT_NOT_FOUND` | 404 | API endpoint doesn't exist |
| `RATE_LIMIT_EXCEEDED` | 429 | Too many requests |
| `INTERNAL_ERROR` | 500 | Server error |
| `BAD_GATEWAY` | 502 | Upstream service error |
| `SERVICE_UNAVAILABLE` | 503 | Temporary service outage |

## Best Practices for Error Handling

<AccordionGroup>
  <Accordion title="Implement Retry Logic">
    For `5xx` errors and `429`, implement exponential backoff:
    
    ```javascript
    async function makeRequestWithRetry(fn, maxRetries = 3) {
      for (let i = 0; i < maxRetries; i++) {
        try {
          return await fn();
        } catch (error) {
          if (error.status >= 500 || error.status === 429) {
            const delay = Math.pow(2, i) * 1000;
            await sleep(delay);
            continue;
          }
          throw error;
        }
      }
    }
    ```
  </Accordion>
  
  <Accordion title="Log Request IDs">
    Always log the `requestId` from error responses for debugging:
    
    ```javascript
    try {
      const response = await talkos.calls.create({...});
    } catch (error) {
      console.error('TalkOS API Error:', {
        code: error.code,
        message: error.message,
        requestId: error.meta?.requestId
      });
    }
    ```
  </Accordion>
  
  <Accordion title="Handle Specific Error Codes">
    Handle errors based on their code, not just HTTP status:
    
    ```javascript
    switch (error.code) {
      case 'INVALID_PHONE_NUMBER':
        // Show user-friendly message about phone format
        break;
      case 'RATE_LIMIT_EXCEEDED':
        // Queue the request for later
        break;
      case 'PLAN_LIMIT_EXCEEDED':
        // Prompt user to upgrade
        break;
      default:
        // Generic error handling
    }
    ```
  </Accordion>
</AccordionGroup>
